# 结构设计

## 文章原型

| 原型 | 核心承诺 | 结构重心 |
|---|---|---|
| 调查实验 | 我替你做了这件事 | 过程、转折、发现 |
| 产品体验 | 跟我一起使用 | 场景、结果、限制 |
| 现象解读 | 这个现象背后是什么 | 观察、追问、解释 |
| 工具分享 | 我发现了一个有用东西 | 问题、试用、用法 |
| 方法论 | 我把可复用经验交给你 | 原则、行动、失败点 |
| 案例复盘 | 这件事为什么成或败 | 决策、结果、因果 |
| 人物故事 | 通过一个人理解议题 | 选择、冲突、代价 |
| 争议评论 | 主流说法遗漏了什么 | 立场、证据、反方 |
| 趋势判断 | 变化正在怎样发生 | 信号、机制、边界 |

## 叙事弧线

- 故事 → 发现 → 原理 → 行动
- 反直觉结论 → 证据 → 反方 → 边界
- 现象 → 疑问 → 调查 → 答案
- 失败 → 复盘 → 原则 → 应用
- 问题 → 现有方案 → 缺陷 → 新解法
- 个人经历 → 公共议题 → 重新理解

原型决定文章任务，弧线决定阅读推进。不要把模板当填空题。

## 非对称权重

为章节分配百分比，总计 100%。主承重章节通常占 30%-45%，过渡章节可以只有 5%-10%。如果所有章节相同长度，检查是否平均用力。

## 承重类比

只有满足以下条件才使用：

1. 移除类比后，文章理解明显变差；
2. 类比至少映射两层机制；
3. 目标读者无需额外课程就能理解；
4. 类比边界已说明，不偷换概念。

### 类比裂口处理

类比"裂开"的地方——即类比与被类比物对不上的地方——往往是文章最值钱的段落。

但**不要因为类比出现裂口就机械宣布失败**，也不要一律"用叙事推过去"。先判断裂口属于哪种类型：

| 裂口类型 | 含义 | 处理方式 |
|----------|------|----------|
| **裂口揭示结论边界** | 裂口本身就是一个有价值的洞察 | 把裂口写出来，作为洞察纳入文章 |
| **裂口只是局部不完全映射** | 类比在大部分维度有效，只是某个细节对不上 | 说明边界后继续使用类比 |
| **裂口破坏核心因果** | 裂口出现在类比的核心映射机制上，导致类比的论证链条断裂 | **必须弃用类比**，换其他论证方式 |
| **裂口会误导读者** | 裂口可能让读者得出与文章意图相反的结论 | **不得靠叙事掩盖**，必须弃用或显式拆解 |

只有前两种可以纳入叙事。后两种是致命裂口，必须弃用类比。

**判断步骤：**
1. 定位裂口：类比和被类比物在哪里对不上？
2. 判断裂口性质：是边界洞察、局部不完整、核心因果断裂、还是误导性？
3. 如果是边界洞察或局部不完整：纳入叙事，展开差异。
4. 如果是核心因果断裂或误导性：弃用类比，不要用叙事硬推。

**补充原则：**
- 完美覆盖一切的类比要么太模糊要么太工整。如果一个类比看似完美覆盖了所有方面，检查它是否只是同义反复。
- 好的承重类比应该有明确的、非致命的裂口——它帮助读者理解大部分机制，同时在裂口处让读者意识到"这里不一样"。

示例：
> 类比："AI 像电力"——映射两层机制：基础设施化（无处不在但不可见）+ 按需取用（插上就用）。
> 裂口：电力是确定性输出（同电压同频率），AI 是概率性输出（同一输入可能不同结果）。
> 判断：裂口揭示结论边界（非致命）。
> 处理：把裂口写出来作为洞察——"电力插上就有电，但 AI 插上不一定有正确答案"。

反例：
> 类比："写作 Skill 像流水线"——映射：输入→处理→输出。
> 裂口：流水线的产品质量取决于机器精度，而文章质量取决于作者判断力，Skill 不能替代判断。
> 判断：裂口破坏核心因果（类比的"标准化产出"核心映射断裂）。
> 处理：弃用类比，改用"工具"或"助手"框架。

## 章节检查

每节必须明确：唯一任务、核心论点、证据、作者经历、与主线关系、篇幅权重。无法填写唯一任务的章节应合并或删除。

## 语义规划（v0.3 新增）

在结构设计阶段，为每节额外规划语义形态，使初稿能生成可映射的结构化内容，formatter 不需要重新猜测文章内容形态。

加载 `references/semantic-components.md`（统一语义词表）和 `references/formatter-capability-map.md`（排版组件能力映射）。

### 每节新增字段

| 字段 | 含义 | 示例 |
|------|------|------|
| content_shape | 本节的内容形态 | structured_comparison / narrative / step_by_step / data_report |
| semantic_blocks | 本节将使用的语义角色 | comparison, key_statement |
| formatter_opportunities | 候选排版组件 | compare, oneliner-card |
| required_payload | 结构化载荷必需字段 | 对象 A/B、对比维度、行数据 |
| fallback_shape | 载荷缺失时的降级形态 | paragraphs + ordered_list |

### 示例

```
## 第二节
- 唯一任务：解释两种工作方式的关键差异
- 核心论点：Agent 与普通提示词的差别不在自动化，而在任务闭环
- content_shape: structured_comparison
- semantic_blocks:
  - comparison
  - key_statement
- formatter_opportunities:
  - compare
  - oneliner-card
- required_payload:
  - 对象 A：普通提示词
  - 对象 B：Agent
  - 对比维度：目标、状态、工具、反馈、退出条件
- fallback_shape: paragraphs + ordered_list
```

### 语义规划规则

1. **先根据内容选择语义形态，再由 formatter 选择具体组件。**
2. **不得为了使用组件改变文章论证。**
3. **同一个信息不得同时完整复制到多个组件。**
4. **一篇文章的主要高级组件建议控制在 3–6 种。**
5. **key_statement 建议全文不超过 5 个。**
6. **表格、对比、时间线、FAQ 必须有完整结构化载荷。**
7. **缺少载荷时必须使用 fallback，不允许 formatter 编造。**
8. **不要要求每篇文章使用全部组件——内容需要时能稳定触发，内容不需要时不强行使用。**
