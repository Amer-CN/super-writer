# Phase 4 评测方案

> 版本：v0.2-rc1 冻结后的第四阶段评测规范
> 日期：2026-07-14
> 状态：待执行

## 一、冻结声明

当前代码冻结为 **super-writer-v0.2-rc1**。

评测期间：
- 不再修改规则
- 不调整评分权重
- 不修测试样本
- 所有发现先登记为 issue
- 完成第一轮后统一形成 rc2

## 二、24 个行为测试任务

### 任务分布

| 类型 | 数量 | 说明 |
|------|------|------|
| 产品体验 | 3 | 实际使用产品后的体验文章 |
| 现象解读 | 3 | 对某个现象的深度分析 |
| 工具分享 | 3 | 介绍工具及其使用方法 |
| 方法论 | 3 | 提出或总结一套可操作的方法 |
| 案例复盘 | 3 | 对项目/事件的事后分析 |
| 观点评论 | 3 | 对某个话题的观点表达 |
| 个人经历型 | 2 | 以第一人称经历为主的文章 |
| 素材不足/故意无法完成 | 2 | 测试系统识别素材不足的能力 |
| 证据冲突/核心崩塌 | 2 | 测试系统发现矛盾并退出的能力 |
| 合计 | 24 | |

### 失败型任务说明

最后四个失败型任务至关重要。优秀写作系统不仅要会写，还要会：
- 不编造事实
- 不强行写
- 发现核心崩塌
- 要求作者补充
- 正确降低交付级别

### 任务设计原则

- 每个任务必须包含**真实素材**（不能只有一句话的抽象选题）
- 素材不足的任务要**确实缺少关键信息**
- 证据冲突的任务要**埋入真实的矛盾证据**
- 每个任务标注预期行为（成功产出 / 降级退出 / 失败退出）

## 三、四版本统一运行

每个任务分别运行：

| 版本 | 说明 |
|------|------|
| A | 裸模型，不加载 Skill |
| B | 原版 khazix-writer |
| C | super-writer v0.1 |
| D | super-writer v0.2-rc1 |

### 固定条件

- 同一模型版本
- 同一系统环境
- 同一素材
- 同一工具权限
- 同一上下文
- 同一输出目标
- 尽可能相同的随机性配置
- 不给某个版本额外解释

### 运行规模

- 24 个任务 × 4 个版本 = **96 个输出**
- 第一轮每个任务只跑一次
- 挑选差异最明显或结果不稳定的 8 个任务，再各跑两次，检查稳定性
- 稳定性检查：8 × 2 × 4 = **64 个额外输出**
- 完整总数：96 + 64 = **160 个输出**

## 四、评测指标

### 硬指标（客观判断）

| 检查项 | 判定标准 |
|--------|----------|
| 是否虚构事实 | 任何虚构 = Hard Fail |
| 是否虚构第一人称经历 | 任何虚构 = Hard Fail |
| 是否正确识别素材不足 | 应退出时未退出 = Hard Fail |
| 是否正确执行失败退出 | 应退出时未退出 = Hard Fail |
| 关键事实是否可追溯 | 不可追溯 = Hard Fail |
| 是否遗漏明显反方证据 | 遗漏 = 扣分项 |
| 是否把推测写成事实 | 出现 = Hard Fail |
| 是否遵守输出模式 | 违反 = 扣分项 |
| 是否保留编辑锚点 | 缺失 = 扣分项 |
| 是否发生职责越界 | 越界 = 扣分项 |
| 是否生成未经要求的 SEO/排版/去 AI 味 | 出现 = 扣分项 |

**任何事实或经历虚构，都应记录为 Hard Fail，不能被其他高分抵消。**

### 软指标（人工盲评）

| 维度 | 分值 | 说明 |
|------|------|------|
| 核心观点 | 15 | 文章是否有清晰、有价值的核心观点 |
| 独特性与信息增量 | 15 | 是否提供了读者无法轻易获得的信息 |
| 论证与证据 | 15 | 论证链条是否完整，证据是否充分 |
| 结构推进 | 15 | 文章结构是否有效推进核心观点 |
| 具体性 | 10 | 是否有具体的案例、数据、细节 |
| 读者价值 | 10 | 读者读完后获得了什么 |
| 作者声音 | 10 | 是否有清晰的作者个人声音 |
| 可读性 | 5 | 是否易读、流畅 |
| 完成度 | 5 | 文章是否完整 |
| 合计 | 100 | |

**去 AI 味不作为本轮主评分项**，因为它属于另一个 Skill。可以记录，但不用于判定写作 Skill 胜负。

## 五、盲评流程

### 匿名化

把四版本输出随机改名：
```
T01-X
T01-Y
T01-Z
T01-W
```

评审时不显示版本来源。

### 评审步骤

每个任务先做两件事：
1. **四篇两两比较**，选最好和最差
2. **再分别评分**（按 9 维度打分）

### 重点观察

- D 对 C 的胜率
- D 对 B 的胜率
- D 的 Hard Fail 数量
- D 是否因为流程复杂反而降低可读性
- D 是否过度研究
- D 是否经常拒绝本来可以写的任务
- D 是否出现规则感、模板感

## 六、发布门槛

```yaml
release_gate:
  v0_2_vs_v0_1_pairwise_win_rate: ">= 65%"
  v0_2_vs_khazix_pairwise_win_rate: ">= 55%"
  hard_fail_rate: "0 次即不通过（24 个任务中任何 1 次 Hard Fail 都不通过）"
  fabricated_experience_count: 0
  boundary_violation_rate: "<= 5%"
  median_content_score_gain_vs_v0_1: ">= 5"

# 注意：24 个任务时 1 次 Hard Fail = 4.17%，因此 3% 阈值实际上意味着 0 次。
# 文档明确：任何一次 Hard Fail 都不通过，不要伪装成精确的 3%。
```

如果 D 只是评分更高，但盲评胜率没有提高，就说明评分器在奖励自己的规则，而不是真正奖励好文章。

## 七、数据集分离

三个数据集严格分离，不混用：

### 1. 系统行为评测集

- 24 个任务
- 用于比较 A/B/C/D 四版本
- 不用于调整评分器权重

### 2. 评分器校准集

- 24 篇有人工标签的文章
- 用于调整评分器权重
- 不用于系统行为评测

### 3. 评分器盲测集

- 12 篇从未参与权重调整的文章
- 用于验证：
  - MAE
  - Spearman
  - 发布/修改/拒绝分类准确率
- **最终有效性以此 12 篇盲测集为准**

不能因为评分器在 24 篇校准集上表现很好，就认为它有效。

## 八、Burstiness 个人基线

### 当前阶段（临时）

- 先用 5～10 篇用户认可的人工文章做临时基线
- Burstiness 只输出诊断
- 不参与总分
- 不阻塞文章

### 积累阶段（≥ 20 篇后）

- 分别计算分文章类型基线：
  - 观点评论
  - 工具教程
  - 故事型文章
  - 方法论文章
- 不能用一个统一基线覆盖所有文章

## 九、需要准备的实际材料

要正式进入第四阶段，需要以下材料：

| 材料 | 用途 | 最低数量 |
|------|------|----------|
| super-writer-v0.2-rc1.zip | 文件级审计 | 1 |

| 包含真实素材的测试主题 | 系统行为评测 | 24 个 |

## 十、执行时间线

| 步骤 | 内容 | 预计工时 |
|------|------|----------|
| 1 | 冻结确认 + 文件审计 | 0.5 天 |
| 2 | 准备 24 个测试任务（含真实素材） | 2 天 |
| 3 | 四版本运行 96 个输出 | 2 天 |
| 4 | 硬指标检查 | 1 天 |
| 5 | 盲评（两两比较 + 评分） | 3 天 |
| 6 | 稳定性检查（8 任务 × 2 次 × 4 版本） | 1 天 |
| 7 | 汇总结果 + issue 分类 | 1 天 |
| 8 | 决策：通过 → v0.2 正式版 / 不通过 → rc2 | 0.5 天 |
